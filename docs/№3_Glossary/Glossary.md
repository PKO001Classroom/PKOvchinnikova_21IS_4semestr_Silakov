### **Глоссарий терминов учебного проекта (УП.01)**

Краткое объяснение ключевых технологий и концепций, используемых в проекте по разработке веб-API с клиентами на разных технологических стеках.

---

#### **Backend-фреймворк: FastAPI**
**Что это?** Современный асинхронный веб-фреймворк на Python для быстрого создания API.
**Ключевая роль в проекте:** Ядро бэкенда. Обрабатывает HTTP-запросы, бизнес-логику, работу с БД.
**Особенности:**
*   **Автодокументация:** Автоматически генерирует интерактивную документацию API (Swagger UI) на основе кода.
*   **Валидация данных:** Через интеграцию с **Pydantic**. Модели описывают, какие данные ожидать и возвращать.
*   **Асинхронность:** Поддержка `async/await` для эффективной обработки множества запросов (например, к БД).
*   **Простота:** Минимум шаблонного кода, чистый и понятный синтаксис.
**В проекте:** Все эндпоинты (`/tasks`, `/users` и т.д.) создаются с помощью декораторов FastAPI (`@app.get`, `@app.post`).

---

#### **Frontend-фреймворк (Вариант А): Nest.JS**
**Что это?** Прогрессивный серверный фреймворк для Node.js, построенный на TypeScript.
**Ключевая роль в проекте:** **Один из вариантов клиентской части.** Используется для создания структурированного веб-приложения, которое взаимодействует с нашим API.
**Особенности:**
*   **Архитектура:** Строгая модульная структура (Контроллеры, Сервисы, Модули) по аналогии с Angular.
*   **Внедрение зависимостей (DI):** Упрощает управление компонентами и их тестирование.
*   **TypeScript "из коробки":** Повышает надежность кода за счет статической типизации.
**В проекте:** Клиентское приложение на Nest.JS отправляет HTTP-запросы (через `axios` или `fetch`) к нашему FastAPI-бэкенду для получения и отправки данных.

#### **ORM-библиотека: SQLAlchemy**
**Что это?** Инструмент (Object-Relational Mapper), который связывает таблицы базы данных с классами Python.
**Ключевая роль в проекте:** Главный посредник между кодом на Python и СУБД PostgreSQL.
**Как работает?** Вместо написания "сырого" SQL вы работаете с объектами:
    ```python
    new_task = Task(title="Изучить FastAPI") # Создаем объект Python
    db.add(new_task) # SQLAlchemy формирует и выполняет INSERT-запрос
    ```
**Преимущества:**
*   **Безопасность:** Защита от SQL-инъекций.
*   **Производительность:** Снижает объем ручного кода.
*   **Гибкость:** Возможность использовать как высокоуровневые операции, так и "сырые" SQL-запросы при необходимости.
**В проекте:** Модели `Task`, `User` и другие описываются как классы SQLAlchemy, которые затем используются в CRUD-операциях.

---

#### **Библиотека валидации: Pydantic**
**Что это?** Библиотека для проверки данных и работы с настройками, основанная на аннотациях типов Python.
**Ключевая роль в проекте:** Гарантия корректности данных, которые приходят в API и уходят из него.
**Как работает?** Вы создаете схемы (модели), описывающие формат данных. FastAPI использует их для:
1.  **Валидации входящих данных** (тело запроса, параметры).
2.  **Сериализации исходящих данных** (преобразование объектов Python/BД в JSON).
3.  **Автодокументации** (схемы отображаются в Swagger UI).
**В проекте:** Создаются `Pydantic-модели` для запроса на создание задачи (`TaskCreate`) и для ответа с задачей (`TaskResponse`).

#### **Клиентская платформа (Вариант Б): Telegram Bot на Aiogram**
**Что это?** **Второй вариант клиента** — Telegram-бот, как интерфейс для взаимодействия с системой.
**Ключевая роль в проекте:** Альтернативный UI. Пользователь общается с ботом, который "под капотом" делает запросы к нашему API.
**Технологический стек:**
*   **Telegram Bot API:** Интерфейс от Telegram для управления ботами.
*   **Aiogram:** Асинхронный Python-фреймворк для удобного создания ботов.
**Как работает?** Пользователь пишет `/newtask` → бот (Aiogram) запрашивает данные → отправляет `POST /tasks` на наш FastAPI-бэкенд → получает ответ → показывает результат пользователю.

---

#### **Драйвер БД: asyncpg**
**Что это?** Быстрая асинхронная библиотека для прямого взаимодействия с PostgreSQL из Python.
**Ключевая роль в проекте:** Низкоуровневый "проводник" для SQLAlchemy, когда она работает в асинхронном режиме.
**Особенности:** Позволяет выполнять SQL-запросы не блокируя основной поток приложения, что критично для производительности асинхронного FastAPI.

---

#### **База данных: PostgreSQL**
**Что это?** Мощная, открытая реляционная система управления базами данных (СУБД).
**Ключевая роль в проекте:** Надежное хранилище для всех данных проекта (задач, пользователей и т.д.).
**Почему именно она?** Соответствует промышленным стандартам, обладает высокой стабильностью, поддерживает сложные типы данных (в т.ч. JSON) и транзакции.
**В проекте:** Структура БД описывается на языке SQL в файле `init_db.sql` на основе ER-диаграммы.

---

#### **Контроль версий: Коммит (Commit)**
**Что это?** Фиксация изменений в коде в системе Git. Каждый коммит — это "снимок" состояния проекта на определенный момент.
**Принципы в проекте:**
*   **Атомарность:** Один коммит = одна логически завершенная задача (например, "добавлена модель Task", "исправлена ошибка в эндпоинте POST").
*   **Соглашение по сообщениям:** Используется `Conventional Commits` для понятной истории (`feat:`, `fix:`, `docs:`, `chore:`).
**Пример:** `git commit -m "feat(api): add GET /tasks endpoint with filtering"`

#### **Уровни тестирования в проекте**

##### **Модульное тестирование (Unit Testing)**
**Цель:** Проверить работу **отдельных изолированных компонентов** (функций, методов, классов).
**Что тестируем?** Логику сервисных функций (например, функцию форматирования даты или расчета рейтинга).
**Инструмент:** `pytest`.
**Как?** Зависимости (БД, внешние API) заменяются **моками (mock)**, чтобы тест был быстрым и изолированным.
> **Неделя 4:** Пишем модульные тесты для ключевых функций бэкенда.

##### **Интеграционное тестирование (Integration / E2E Testing)**
**Цель:** Проверить **взаимодействие всех компонентов** системы от начала до конца.
**Что тестируем?** Полный сценарий: "Клиент → API → База данных → API → Клиент".
**Пример сценария:**
1.  Тест-клиент отправляет `POST /tasks` с данными новой задачи.
2.  Проверяется, что в БД появилась соответствующая запись.
3.  Проверяется, что в ответ пришел корректный JSON с ID созданной задачи.
> **Неделя 4:** Пишем интеграционные тесты для основных сценариев работы API.
